# 项目级 Codex 指南

适用范围：本文件所在目录及所有子目录。若更深层目录出现新的 `AGENTS.md`，以更深层说明为准。

## 项目定位

- 本项目是本地运行的 Python CLI，用于医学、养生、健康科普类小红书内容的调研、审核、润色、格式检查、发布包生成和发布辅助。
- 当前发布能力是 `ManualPublisher`：只辅助用户手动复制最终发布包，不登录小红书，不访问平台发布接口，不保存账号密码。
- 后续自动登录发布、浏览器自动化或官方 API 发布能力必须作为可替换发布通道扩展，不得破坏现有调研、审核、打包主流程。

## 语言和文档

- 面向用户的说明、`README.md`、恢复手册和项目文档默认使用中文。
- 代码标识符、包名、命令名、API 名称、配置字段、错误类型保持英文。
- 修改 Markdown、TOML、JSON、YAML、Python 等文本文件时，在 Windows 上优先用 PowerShell 7 并显式指定 UTF-8；如果中文输出乱码，必须重新用 UTF-8 读取后再判断。

## Python 和依赖管理

- 本仓库使用 `uv`、`pyproject.toml` 和 `uv.lock` 管理依赖与命令运行。
- 运行项目命令使用 `uv run ...`，例如 `uv run xhs-agent --help`、`uv run pytest`。
- 同步依赖使用 `uv sync`。
- 添加生产依赖前先征得用户同意；添加开发依赖前说明原因，并优先复用现有工具。
- 不使用 `pip install` 修改项目环境，除非用户明确要求或仓库依赖体系发生变化。

## 常用检查命令

```powershell
uv run ruff format .
uv run ruff check .
uv run mypy src
uv run pytest
```

- 只改文档时可不跑完整测试，但提交前至少检查相关 Markdown 内容和 `git diff`。
- 改 CLI、配置、状态机、审核、发布包或发布通道时，优先运行完整四项检查。
- 新增功能或修复 bug 应补充贴近变更面的单元测试或集成测试。

## 架构约定

- CLI 入口在 `src/xiaohongshu_auto_publish/cli.py`，使用 Typer。
- 应用装配在 `src/xiaohongshu_auto_publish/app.py`；新增服务优先通过装配层接入，不在 CLI 中堆业务逻辑。
- 核心状态和数据模型在 `models.py`，状态流转在 `orchestration/`，持久化在 `state/` 和 `artifacts/`。
- 产物必须版本化写入 `workspace/<task_id>/`，不要覆盖旧版本；状态更新、确认、回滚、重试、发布尝试和失败必须写审计记录。
- `workspace/`、`.env`、`.venv` 是本地数据或秘密相关内容，不应提交到 git。
- 配置字段应先进入 `config/schema.py` 和 `config/loader.py`，再由业务模块读取；不要在业务代码里散落硬编码环境变量读取。

## 医学和平台安全边界

- `--yes` 只能跳过非阻断确认点，不能绕过 S0/S1、结构化解析失败、格式 `block`、素材复验失败或发布确认。
- 不得把 API Key、账号凭据、会话 token、Cookie、验证码、手机号、身份证明等敏感信息写入代码、Markdown 产物、JSONL 日志、错误输出或测试快照。
- 医学内容审核、来源日期未知标记、格式阻断和发布包 `can_publish` 判断必须保持可追溯。
- 任何发布能力都不能替代医生、药师、营养师、合规人员或用户自己的最终判断。

## 自动登录发布和浏览器自动化扩展

- 自动发布必须只读取已生成且 `can_publish=true` 的最终发布包，并要求用户明确确认。
- 浏览器自动化必须支持人工登录和人机验证人工介入，不得绕过验证码、风控、频率限制或平台安全机制。
- 不保存明文账号密码。若需要保存浏览器会话状态，只能放在 `.gitignore` 覆盖的本地目录中，并在文档中说明风险、位置和清理方式。
- 浏览器截图、HTML 快照、失败日志和调试轨迹默认写入任务工作区或其他 ignored 目录，并对敏感信息做脱敏。
- 自动化实现应放在独立发布通道中，例如 `src/xiaohongshu_auto_publish/publish/` 下的新模块，并实现现有 Publisher 协议；不要让主编排流程依赖具体浏览器库。
- 浏览器选择器、等待策略、重试策略和失败恢复要有清晰错误信息；页面改版时应失败为可诊断状态，并能回退到 `ManualPublisher`。
- 浏览器自动化测试应优先使用 fake page、本地 fixture 或 mock，不依赖真实小红书账号、真实网络发布或真实验证码。
- 若引入 Playwright、浏览器驱动或官方 API SDK，先说明依赖用途、运行成本和安全影响，再按 `uv` 工作流添加。

## Bug 修复工作流

- 先复现或定位失败状态，再做最小范围修改。
- 修复状态机、审核或发布包问题时，同时检查审计日志、产物索引和用户可恢复路径是否仍然正确。
- 不要为了让测试通过删除或重写用户工作区数据；需要修改样例任务时，优先新增测试 fixture。
- 保留用户已有改动，不执行 `git reset --hard`、`git checkout --` 等破坏性命令，除非用户明确要求。

## Git 提交

- 提交前查看 `git status --short` 和 `git diff --stat`。
- 提交信息用简洁中文或英文均可，优先说明用户可见变化。
- 用户要求“提交所有未提交修改”时，包含当前工作区中所有已跟踪和未跟踪且未被 `.gitignore` 忽略的文件；否则只提交与任务直接相关的文件。
