# 小红书医学/养生科普 Agent 需求文档

版本：v0.1  
日期：2026-06-04  
状态：需求草案  

## 1. 文档定位

本文档描述“小红书医学/养生科普 Agent”的产品需求、边界、核心流程、功能要求与验收标准。

本文档不是概要设计文档，也不是详细设计文档。涉及模块边界、接口形态、目录结构、具体技术实现、测试用例细节等内容，只在需求层面说明约束，不展开设计。

## 2. 背景与目标

用户希望构建一个本地运行的 Agent，用于辅助完成医学、养生、健康科普类小红书内容的调研、审核、润色、格式检查，并在条件成熟后支持自动发布到小红书。

该 Agent 需要支持两类起点：

1. 用户只给出一个大方向，Agent 进行联网调研并整理信息。
2. 用户已经写好文章，Agent 直接进入审核、润色、格式检查和发布准备流程。

核心目标：

1. 降低医学/养生内容创作的信息检索和整理成本。
2. 尽量降低医学事实错误、危险建议、夸大宣传、平台违规等风险。
3. 让最终内容更适合小红书平台传播，提高点击率、关注转化、账号人设稳定性和系列化运营能力。
4. 通过本地 Markdown 文件流转，保留用户手动修改和最终确认权。
5. 为后续自动发布能力保留可替换、可扩展的发布通道。

## 3. 目标用户

主要用户是希望在小红书持续发布医学、养生、健康科普内容的个人创作者或小团队。

用户可能不熟悉 Agent 构建、LLM 工作流、医学内容审核流程或小红书自动化发布实现，因此系统应尽量通过明确的阶段产物、审核结论和下一步提示降低使用门槛。

## 4. 范围

### 4.1 本阶段范围

本阶段需要支持：

1. 根据大方向进行联网调研。
2. 将调研信息整理为 Markdown 文件。
3. 允许用户在本地手动编辑阶段性 Markdown 文件。
4. 对用户编辑后的内容进行医学/健康内容审核。
5. 将所有审核问题按严重程度直接展示给用户。
6. 对严重问题进行阻断，要求用户修改后重新审核。
7. 内容审核通过后，进行面向小红书的写作审核与润色。
8. 写作审核通过后，进行小红书发布格式审核。
9. 生成最终可发布文案及发布素材检查结果。
10. 支持方便切换 LLM 供应商、模型、`base_url` 和 `api_key`。
11. 将自动发布能力作为扩展能力纳入需求，但不要求本阶段完成具体发布实现。

### 4.2 非本阶段范围

本阶段不要求：

1. 实现完整的小红书自动登录和自动发帖功能。
2. 实现图像生成、图片排版或封面自动设计。
3. 替代医生、药师、营养师等专业人员进行诊断、治疗或处方判断。
4. 保证内容一定获得高曝光、高互动或涨粉。
5. 编写概要设计文档或详细设计文档。

## 5. 总体流程

### 5.1 从大方向开始

1. 用户输入选题大方向、目标受众、期望风格和可选账号定位信息。
2. Agent 联网调研，优先检索权威来源。
3. Agent 生成调研 Markdown 文件，包含核心结论、来源列表、可信度说明和可用素材点。
4. 用户本地手动修改调研文件。
5. Agent 读取修改后的文件并进行内容审核。
6. 如果存在严重错误，Agent 阻断流程，展示全部问题并要求用户修改。
7. 如果不存在严重错误，Agent 展示全部问题，并允许轻微问题带着提示继续。
8. Agent 进行小红书写作审核与润色，生成优化后的文案。
9. 用户可再次手动修改文案。
10. Agent 进行格式审核。
11. 格式审核通过后，生成最终发布包。
12. 后续可通过发布扩展能力登录小红书并发布。

### 5.2 从已有文章开始

1. 用户提供已有文章 Markdown 文件或文本内容。
2. Agent 直接进入内容审核。
3. 后续流程与“从大方向开始”的第 6 步之后一致。

## 6. 阶段产物要求

Agent 应通过本地 Markdown 文件保存阶段性结果，方便用户手动编辑。

建议至少包含以下产物类型：

1. 调研资料文件：保存调研结论、来源、证据等级、可用素材点和不确定信息。
2. 用户编辑稿：用户在本地手动修改后的内容。
3. 内容审核报告：保存医学事实、风险表述、平台合规等问题。
4. 润色稿：保存面向小红书传播优化后的版本。
5. 格式审核报告：保存标题、正文、标签、图片、封面等发布格式检查结果。
6. 最终发布包：保存最终文案、标签、素材清单、审核状态和发布时间记录。

文件命名应便于追踪同一选题的不同阶段和版本。

## 7. 功能需求

### FR-001 选题输入

系统应允许用户输入一个大方向，例如“春季养肝”“高血压人群低盐饮食”“睡眠不足与代谢问题”等。

输入信息应支持但不强制包含：

1. 目标受众。
2. 账号定位。
3. 内容风格。
4. 期望篇幅。
5. 是否需要系列化选题。
6. 是否已有参考资料或已有文章。

### FR-002 联网调研

系统应默认允许联网检索，并记录检索时间。

调研应优先使用以下来源：

1. 国家或地区卫生健康主管部门。
2. 权威医学指南。
3. 公立医院、大学医学中心或专业医学机构。
4. 学术期刊、系统综述、指南共识。
5. WHO、CDC 等国际公共卫生机构。
6. 其他可信健康科普来源。

对于每条关键结论，系统应尽量提供：

1. 来源链接。
2. 来源机构。
3. 发布日期或更新日期。
4. 可信度说明。
5. 与选题的相关性说明。
6. 是否存在争议或证据不足。

如果来源缺少发布日期，系统应明确标记“发布日期未知”，不能伪造日期。

### FR-003 调研结果整理

调研结果应以 Markdown 格式保存并展示。

调研文件应至少包含：

1. 选题摘要。
2. 适合小红书表达的核心观点。
3. 可引用事实。
4. 需要谨慎表达的内容。
5. 不建议使用或风险较高的说法。
6. 来源清单。
7. 适合继续写作的内容结构建议。

### FR-004 本地手动修改

系统应在关键阶段暂停，提示用户可以手动编辑对应 Markdown 文件。

用户完成修改后，应能让 Agent 继续读取该文件并进入下一阶段。

系统不应要求用户只能在聊天窗口中修改内容。

### FR-005 已有文章导入

系统应支持用户直接提供已有文章。

已有文章可以来自：

1. 本地 Markdown 文件。
2. 用户在对话中粘贴的文本。
3. 后续扩展的其他文本来源。

系统应将已有文章规范化为本地 Markdown 阶段产物，再进入审核流程。

### FR-006 内容审核

内容审核应在医学事实、健康风险、平台合规和用户误导风险四个方向进行。

所有发现的问题，无论轻重，都必须直接展示给用户。

内容审核结果应至少包含：

1. 问题位置。
2. 原文片段。
3. 问题类型。
4. 严重程度。
5. 风险说明。
6. 修改建议。
7. 是否阻断流程。

### FR-007 严重问题阻断

如果内容中出现严重错误，系统必须阻断流程，要求用户重新修改后再次审核。

严重问题至少包括：

1. 医学事实错误。
2. 危险用药或治疗建议。
3. 夸大疗效。
4. 暗示或鼓励替代正规治疗。
5. 误导性禁忌。
6. 违反平台医疗健康内容规则的表达。
7. 无证诊疗表述。

系统不得在严重问题未解决时进入写作润色、格式审核或发布阶段。

### FR-008 轻微问题处理

对于非重大错误，系统可以允许流程继续，但必须向用户展示问题和建议。

轻微问题可包括：

1. 表述不够严谨但不会造成明显健康风险。
2. 来源引用不完整但核心事实可追溯。
3. 术语解释不够清晰。
4. 语气偏绝对但风险可通过改写降低。
5. 局部表达不符合账号风格但不构成事实错误。

系统应清楚标记这些问题为“不阻断”。

### FR-009 写作审核与润色

内容审核通过后，系统应进行面向小红书平台的写作审核与润色。

写作优化目标优先级为：

1. 提高点击率。
2. 提高关注转化。
3. 稳定账号人设。
4. 支持系列化选题。

系统还应综合考虑：

1. 提高开头吸引力。
2. 提高正文完读率。
3. 增强收藏、评论、转发动机。
4. 控制标题党和夸张表达风险。
5. 保持医学健康内容的可信度。
6. 降低平台违规风险。
7. 保留用户原始观点和表达边界。
8. 生成适合后续复用的账号表达风格。

默认写作风格为“通俗种草”。系统还应支持切换为：

1. 专业严谨。
2. 平衡风格。

### FR-010 写作审核输出

写作审核应输出：

1. 润色后的标题候选。
2. 润色后的正文。
3. 开头优化说明。
4. 关注引导建议。
5. 互动引导建议。
6. 账号人设一致性评价。
7. 可扩展的系列化选题建议。
8. 被修改的关键表达说明。
9. 仍需用户确认的表达。

系统不能为了流量目标引入新的医学事实错误或夸大承诺。

### FR-011 格式审核

写作审核通过后，系统应进行小红书发布格式审核。

格式审核应覆盖：

1. 标题格式。
2. 正文段落结构。
3. 话题标签。
4. 敏感词或高风险表达。
5. 表情符号使用。
6. 图片数量。
7. 封面标题。
8. 图片比例。
9. 图片与正文的一致性。
10. 发布前必填项。

小红书平台规则可能变化，因此格式审核规则应支持配置和更新，不能只依赖写死在代码中的固定规则。

### FR-012 格式修改确认

如果格式审核发现需要修改的问题，系统不得在用户不知情的情况下直接发布。

系统应支持两种处理方式：

1. 输出修改建议，等待用户手动修改。
2. 在用户明确授权或配置允许的情况下，自动生成格式修复稿，并要求用户在发布前确认。

### FR-013 最终发布包

格式审核通过后，系统应生成最终发布包。

最终发布包应至少包含：

1. 最终标题。
2. 最终正文。
3. 话题标签。
4. 图片素材清单。
5. 封面标题。
6. 格式审核结论。
7. 内容审核结论。
8. 来源清单。
9. 用户确认状态。
10. 是否允许进入发布阶段。

### FR-014 自动发布扩展能力

系统应将登录小红书并自动发帖作为扩展能力设计进需求范围，但本阶段不要求实现。

自动发布能力应采用可替换发布通道的形式，避免将业务流程与某一种发布方式强绑定。

需求层面的最佳实践要求：

1. 发布能力应抽象为独立的发布通道。
2. 发布前必须复用最终发布包和审核状态。
3. 发布前必须确认用户已授权发布。
4. 发布通道应支持后续替换，例如官方 API、浏览器自动化、第三方工具或其他可行方式。
5. 如果使用浏览器自动化，应支持人工登录、人机验证人工介入、会话安全保存和失败重试。
6. 不应绕过平台安全机制、风控机制或验证码。
7. 不应保存明文账号密码。
8. 应记录发布结果、失败原因和可重试状态。

### FR-015 LLM 供应商配置

系统应支持方便修改 Agent 内部 LLM 的供应商和密钥。

首要支持：

1. OpenAI。
2. DeepSeek。

同时应支持通用配置：

1. `provider`。
2. `base_url`。
3. `model`。
4. `api_key`。
5. 可选的 `timeout`。
6. 可选的 `max_retries`。

密钥不应写死在代码或文档正文中。

### FR-016 配置安全

系统应支持通过环境变量或本地配置文件读取密钥。

本地配置文件如包含密钥，应默认不提交到版本控制。

系统应在缺少必要配置时给出清晰错误提示，例如缺少 `OPENAI_API_KEY` 或 `DEEPSEEK_API_KEY`。

### FR-017 审核记录

系统应保存关键审核记录，方便用户回溯。

审核记录应包含：

1. 审核时间。
2. 输入文件路径。
3. 使用的模型。
4. 审核结论。
5. 阻断问题列表。
6. 非阻断问题列表。
7. 用户是否确认继续。

### FR-018 状态提示

系统应在每个阶段给出明确状态提示。

状态至少包括：

1. 等待用户编辑。
2. 等待用户确认。
3. 内容审核未通过。
4. 内容审核通过但有轻微问题。
5. 写作审核完成。
6. 格式审核未通过。
7. 格式审核通过。
8. 可进入发布阶段。
9. 发布成功。
10. 发布失败。

## 8. 内容审核分级

建议使用以下严重程度分级：

| 等级 | 名称 | 是否阻断 | 说明 |
| --- | --- | --- | --- |
| S0 | 严重 | 是 | 可能造成健康风险、事实严重错误、违规或无证诊疗风险 |
| S1 | 高风险 | 是 | 可能误导用户，需修改后才能继续 |
| S2 | 中风险 | 否，默认提示 | 表述不严谨或证据不足，但可通过提示继续 |
| S3 | 轻微 | 否 | 风格、清晰度、来源完整性等轻微问题 |

阻断等级至少包括 S0 和 S1。

## 9. 小红书写作要求

系统应将小红书内容视为“平台化内容”，而不是普通科普文章。

写作审核应重点检查：

1. 标题是否能在不夸大的前提下提高点击意愿。
2. 开头是否能快速说明痛点、场景或反常识点。
3. 正文是否适合移动端阅读。
4. 内容是否有明确账号人设。
5. 是否自然引导关注。
6. 是否适合形成系列内容。
7. 是否有收藏价值。
8. 是否避免绝对化医学承诺。
9. 是否避免制造焦虑或恐慌。
10. 是否保留必要的免责声明或就医提醒。

## 10. 小红书格式要求

格式审核应基于可更新的规则库。

规则库应支持检查：

1. 标题长度、符号、敏感表达和可读性。
2. 正文长度、换行、段落密度和列表结构。
3. 话题标签数量、相关性和风险词。
4. 表情符号密度和使用位置。
5. 封面标题长度和可读性。
6. 图片数量是否满足发布需求。
7. 图片比例是否适合小红书展示。
8. 图片内容是否与正文一致。
9. 是否存在不适合医学健康内容的营销化表达。
10. 是否存在平台可能限制或拒绝发布的高风险词。

当平台规则变化时，应能更新规则库，而不需要重写完整 Agent 流程。

## 11. 自动发布扩展要求

自动发布能力应被视为独立扩展，不应影响调研、审核、润色和格式检查的主流程。

发布扩展至少应满足：

1. 只接受已通过审核的最终发布包。
2. 发布前要求用户确认。
3. 支持人工登录或用户授权。
4. 支持记录发布页面、发布时间、结果截图或失败日志。
5. 支持失败后重试或回退到手动发布。
6. 支持未来替换为更稳定的发布方式。

如小红书提供官方开放 API，应优先考虑官方 API。若官方 API 不可用，浏览器自动化可以作为候选扩展方式，但必须遵守平台规则和用户授权边界。

## 12. 非功能需求

### 12.1 可用性

系统应让不了解 Agent 构建的用户也能通过清晰的文件和状态提示完成流程。

### 12.2 可追溯性

所有关键事实、审核结论和发布决策都应可追溯到阶段文件或审核记录。

### 12.3 可配置性

LLM 供应商、模型、API 地址、密钥、审核阈值、写作风格和格式规则应可配置。

### 12.4 安全性

系统不得明文硬编码 API Key、账号密码或其他敏感凭据。

系统不应绕过平台安全机制。

### 12.5 稳定性

联网调研、模型调用和发布扩展都可能失败。系统应给出可理解的失败原因，并允许用户重试。

### 12.6 合规性

系统应尽量降低医学健康内容误导风险和平台违规风险，但不应声称可以替代专业医疗审核或法律合规审核。

## 13. 配置需求

系统应至少支持以下配置项：

```env
LLM_PROVIDER=openai
LLM_BASE_URL=https://api.openai.com/v1
LLM_MODEL=
LLM_API_KEY=
LLM_TIMEOUT=
LLM_MAX_RETRIES=
```

DeepSeek 可通过同一套配置方式切换：

```env
LLM_PROVIDER=deepseek
LLM_BASE_URL=https://api.deepseek.com
LLM_MODEL=deepseek-v4-flash
LLM_API_KEY=
```

具体配置文件名称、加载优先级和密钥存储方式留待后续设计文档确定。

## 14. 验收标准

### AC-001 大方向调研

给定一个健康科普大方向，系统能够联网调研并生成 Markdown 调研文件，文件中包含来源链接、来源机构、发布日期或未知标记、可信度说明。

### AC-002 已有文章审核

给定一篇已有文章，系统能够跳过调研阶段，直接生成内容审核报告。

### AC-003 手动编辑续跑

用户手动修改阶段 Markdown 文件后，系统能够读取修改后的内容并继续后续流程。

### AC-004 严重问题阻断

当文章包含危险用药建议、夸大疗效或替代正规治疗等严重问题时，系统必须阻断流程，并展示全部问题。

### AC-005 轻微问题展示

当文章只存在轻微问题时，系统可以允许继续，但必须展示所有轻微问题和修改建议。

### AC-006 写作润色

内容审核通过后，系统能够输出符合所选风格的小红书润色稿，并说明标题、开头、关注引导和系列化选题相关优化。

### AC-007 格式审核

系统能够检查标题、正文、话题标签、敏感词、表情符号、图片数量、封面标题、图片比例等格式要求，并输出是否可发布。

### AC-008 发布前确认

在自动发布扩展启用时，系统不得在未经用户确认的情况下发布内容。

### AC-009 LLM 配置切换

用户能够通过配置切换 OpenAI、DeepSeek 或自定义兼容供应商，而无需修改核心业务代码。

### AC-010 密钥安全

API Key 不应出现在源码、需求文档示例值或版本控制记录中。

## 15. 风险与限制

1. 医学健康内容具有高风险，Agent 审核不能替代专业医生、药师、营养师或合规人员审核。
2. 联网来源可能过期、失效或质量不稳定，系统必须标记来源和检索时间。
3. 小红书平台规则和展示机制可能变化，格式规则应可更新。
4. 自动发布可能受登录、验证码、平台风控、页面改版影响。
5. 流量目标无法保证，只能通过内容结构、标题、账号定位和运营策略提高概率。
6. LLM 可能产生幻觉，因此关键事实必须有来源支撑。

## 16. 待确认项

以下事项不影响本需求文档成立，但需要在后续概要设计或详细设计前进一步确认：

1. 阶段文件的具体目录结构和命名规范。
2. 是否需要支持多账号、多账号人设和账号内容库。
3. 是否需要为每篇内容生成配图建议、封面文案或图片生成提示词。
4. 是否需要接入固定医学数据库或指定网站白名单。
5. 是否需要人工专家复核流程。
6. 是否需要支持定时发布、草稿箱管理和发布后数据回收。
7. 是否需要保存完整的选题库、系列规划和历史内容复盘。
